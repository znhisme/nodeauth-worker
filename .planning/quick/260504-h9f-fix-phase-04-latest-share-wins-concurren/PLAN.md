# Quick Task: Fix Phase 04 latest-share-wins concurrency security gap

## Objective

Close the Phase 04 security blocker where concurrent owner share creation can leave multiple active links for the same owner and vault item.

## Plan

1. Add a database-enforced active-share guard using a nullable `active_share_key`.
2. Clear `active_share_key` when old active shares are revoked.
3. Insert new shares with `active_share_key = ownerId:vaultItemId`.
4. Retry replacement on unique-conflict insert failures so the latest request wins.
5. Add focused repository/service regression tests for the conflict path.
6. Regenerate Worker, Docker, and Netlify backend bundles.
7. Re-run Phase 04 security verification.

## Files In Scope

- `src/shared/db/schema/*.ts`
- `src/shared/db/migrator.ts`
- `backend/schema.sql`
- `src/shared/db/repositories/shareRepository.ts`
- `src/features/share/shareService.ts`
- share repository/service tests
- backend dist bundles generated by build scripts
- Phase 04 GSD security artifacts

## Success Criteria

- Concurrent/conflicting create attempts cannot leave multiple active owner+vault share rows.
- Phase 04 open security threats for latest-share-wins close.
- Backend tests and supported runtime builds pass.

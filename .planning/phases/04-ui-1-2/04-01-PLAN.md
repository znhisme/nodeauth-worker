## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner client -> owner share service | Authenticated owner input chooses which vault item to share and replacement timing. |
| share service -> database | Service writes revocation state and audit events for sensitive share records. |
| public recipient -> public share service | Old raw public tokens may be retried after replacement. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `ShareService.createShare` / `ShareRepository.revokeActiveForOwnerVaultItem` | mitigate | Revoke active owner+vault shares server-side before returning the new share; test old-token public access resolves generic inaccessible. |
| T-04-02 | Information Disclosure | replacement audit metadata | mitigate | Audit metadata contains only `revokedAt` and `reason: "latest_share_wins"`; tests assert no raw token/code, hashes, password, secret, seed, or full URL. |
| T-04-04 | Information Disclosure | public recipient response for old token | mitigate | Replaced shares use the existing revoked/inaccessible public decision with no-store/no-referrer headers and no condition-specific public error. |
| T-04-05 | Tampering | frontend generated chunks | accept | This plan touches backend source only and does not edit `frontend/dist/**`; UI work is gated in Plan 03. |

# Plan 04: Close Share Management UI UAT Gap

## Objective

Implement the missing user-visible Phase 4 share management UI and batch sharing entry while preserving the generated-output boundary by introducing editable frontend source and a repeatable build script.

## Tasks

1. Restore editable frontend source for the share UI overlay.
   - Add `frontend/package.json`.
   - Add source files under `frontend/src/share-ui/`.
   - Build output must be generated into `frontend/dist/assets/share-management-ui.js`.

2. Add actual UI behavior.
   - Add a `Manage Shares` left-navigation entry after login.
   - Open a management view that fetches `GET /api/share`.
   - Display only owner-safe share metadata.
   - Allow revoking active shares through `DELETE /api/share/:id`.
   - Add a Share button between Delete and Cancel when accounts are selected.
   - Batch share selected account ids through `POST /api/share/batch`.
   - Show one-time public URL/access-code results without browser storage persistence.

3. Wire built output into the deployed SPA shell.
   - Add source-controlled build script.
   - Reference generated CSS/JS from `frontend/dist/index.html`.

4. Verify.
   - Run the frontend build script.
   - Assert `frontend/dist` contains `Manage Shares`, `Delete`, `Share`, `Cancel`, and `/api/share/batch`.
   - Run backend tests for share routes.
   - Use browser automation to confirm the injected UI entry points render.

## Acceptance

- Incognito login users can see a share management entry after authentication.
- Share management opens an owner-safe management view.
- Selecting accounts in My Accounts shows `Delete`, `Share`, `Cancel` in that order.
- Batch share calls the backend batch route and displays one-time results.
- Raw tokens/access codes are not written to localStorage, IndexedDB, service-worker cache, route state, or query params by this UI.

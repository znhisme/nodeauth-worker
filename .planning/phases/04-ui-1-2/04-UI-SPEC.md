# Phase 4 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 4 defines the desired owner UI for share management and batch sharing. Editable frontend source is absent in this checkout; only generated `frontend/dist/**` Vue/Vite/Element Plus assets are present. The planner must not use hand edits to generated Vite chunks as the primary delivery path. UI implementation is gated on restoring editable Vue frontend source or an equivalent maintainable frontend source tree.

The backend must enforce one effective share per account and latest-share-wins before the UI claims that behavior. The UI may display only the latest active share per account as primary, but it must not hide still-usable older shares as a substitute for server enforcement.

| Surface | Required Interaction Contract | Implementation Constraint |
|---------|-------------------------------|---------------------------|
| Left navigation | Add a first-level authenticated owner navigation item labeled `Manage Shares`. It opens the share management page and follows the existing `app_active_tab` left-menu pattern. | Requires restored frontend source; generated `frontend/dist/assets/index-*.js` is evidence only. |
| Share management page | Show accounts that have share records, grouped by account item, with current effective share first and historical expired/revoked records available as safe metadata. | Consume owner share metadata only; do not request or display passwords, OTP seeds, raw tokens, access-code hashes, owner identity, sessions, backups, or unrelated vault lists. |
| Single-account create/regenerate | From an account row, `Create share link` creates a share. If an active share already exists for that account, the new share becomes the only effective share and older active shares are revoked/expired by backend logic. | Raw public URL and access code are shown exactly once in a post-create dialog. They are not recoverable from later details. |
| Single-account revoke | Owner can revoke the current active share from the management page or account row after confirming the revocation limitation. | Revoke copy must state that future access is blocked but copied credentials cannot be retracted. |
| My Accounts bulk selection | Preserve existing select-all-loaded and group-select behavior. When one or more non-deleted accounts are selected, add a `Share` button between `Delete` and `Cancel`. | Selection is client-side, but ownership, latest-share-wins, and one-time secret creation are backend responsibilities. |
| Batch share result | Batch sharing returns per-account results. Successfully created shares show public URL and access code exactly once with copy controls. Failed rows show safe item label and generic owner-safe failure. | Do not persist raw URLs or access codes to local storage, IndexedDB, service-worker cache, logs, or route/query params. |
| Offline state | Share creation, batch sharing, revoke, and recipient unlock are online-only. | Disable actions when physical offline or manual offline mode is active; do not enqueue share operations in offline sync. |
| Public recipient page | Existing public share behavior remains generic and privacy-preserving. This phase does not add a public vault browser. | Public failure states remain generic for missing, expired, revoked, deleted, locked, wrong-code, and malformed-token conditions. |

| Body | 16px | 400 | 1.5 |
| Heading | 20px | 600 | 1.2 |

Use only weights 400 and 600. Use 16px text for mobile inputs, including access-code and expiration inputs, to avoid browser zoom. Secret-like values such as share URLs and access codes may use the existing monospace treatment, but stay within this size and weight scale. Long URLs must wrap or truncate in the middle with a copy button; they must not overflow dialogs, drawers, tables, or mobile screens.

| Status | Color Token | Usage |
|--------|-------------|-------|
| Active | `var(--el-color-success, #67c23a)` | The current effective share can be used until expiration or revocation |
| Expired | `var(--el-color-info, #909399)` | TTL ended; link cannot be used again |

Every status must include text plus icon or tag; color alone is insufficient.

## Interaction Contract

### Navigation And Layout

1. Add `Manage Shares` as an authenticated left-side menu item near `My Accounts`/vault functions, not under advanced settings.
2. Desktop layout uses the existing left sidebar, page header, `tab-card-wrapper`, and Element Plus table/list/card density. Do not create a marketing-style page or oversized hero.
3. Mobile layout follows the existing subpage behavior with a header title and back action. Use a bottom drawer for one-time share results if the existing source pattern supports it; otherwise use an Element Plus dialog sized to the viewport.
4. The page header contains title, short description, refresh action, and an optional status filter. Filters use segmented controls or Element Plus tabs: `All`, `Active`, `Expired`, `Revoked`.
5. List density must support repeated management work: account label, status, expiration, last accessed, access count, and row actions are visible without opening a detail panel on desktop.

### Share Management Page

1. Fetch owner share metadata from the owner API. The UI renders only `id`, `item.service`, `item.account`, `status`, `createdAt`, `expiresAt`, `revokedAt`, `lastAccessedAt`, `accessCount`, and `publicUrl` if it is returned at creation time.
2. Group rows by account item. The current effective share appears first for each account. Historical expired/revoked records are visually subordinate.
3. Active rows show `Active`, expiration time, last accessed time or `Never accessed`, and access count.
4. Expired rows show `Expired`, expiration time, last accessed time, and access count.
5. Revoked rows show `Revoked`, revoked time, last accessed time, and access count.
6. The page must not imply that revocation or expiration retracts credentials already copied.
7. The page must not provide a "show access code" action for existing shares. Access codes are one-time creation outputs.
8. Revoke is available only for the current active share. Historical revoked/expired rows do not show destructive actions.
9. Refresh reloads metadata and clears any selected row state, but never rehydrates raw URL or access-code values.

### Single-Account Create And Regenerate

1. `Create share link` creates a share for exactly one account item.
2. If the account already has an active share, the confirmation copy must say a new link replaces the current active link and the old link stops working.
3. Backend latest-share-wins is required. The UI must not simulate replacement by hiding older active rows.
4. The post-create dialog displays service/account label, expiration, public URL, access code, `Copy link`, `Copy access code`, and `Copy both`.
5. The dialog must include: `Copy the link and access code now. NodeAuth cannot show this access code again.`
6. Closing the dialog discards raw public URL and access code from component state.
7. The UI must not store raw URL, raw token, access code, password, OTP code, or OTP seed in browser storage, analytics, logs, route state, or query params.

### My Accounts Bulk Share

1. Preserve existing account selection model. `Select all` means all loaded accounts in the current view. Group select-all means all loaded accounts in that group unless a backend all-matching selection endpoint is added.
2. When one or more non-deleted accounts are selected, show toolbar actions in this order: `Delete`, `Share`, `Cancel`.
3. `Share` is disabled when offline, when only deleted/trash rows are selected, or while a batch request is in flight.
4. The button label on desktop is `Share`; the accessible name is `Share selected accounts`. On mobile, icon-only is allowed only with the same accessible name and tooltip.
5. The batch confirmation shows the selected count and warns: `Creating new links will replace any current active links for selected accounts. Old links will stop working.`
6. Batch result rows show account label, status, expiration, public URL, access code, and copy controls for successes. Failed rows show safe account label if owner-scoped and a generic failure reason such as `Could not create link`.
7. Batch results support `Copy all links and codes` for successful rows only.
8. Partial failures must not discard successful raw secrets before the owner can copy them.
9. After the result dialog closes, clear raw secrets from component state, refresh share metadata, and keep account selection behavior consistent with existing bulk delete semantics.

### Recipient Flow

1. Public recipient access remains unauthenticated and requires access code submission in a request body or equivalent non-URL channel.
2. Recipient success displays only `SharedItemView`: service, account, password if included, and current OTP code/countdown if available.
3. Recipient failure states use the generic unavailable heading and body for missing, invalid, expired, revoked, deleted, locked/rate-limited, wrong-code, cleanup-affected, and malformed-token cases.
4. Public recipient views must not expose owner identity, vault lists, unrelated account counts, session state, backup metadata, raw internal IDs, raw token, access-code hash, OTP seed, otpauth URI, or API diagnostics.
5. Public recipient views must not be cached by service worker behavior and must preserve backend no-store/no-referrer protections.

### API Response Semantics For UI

| API Outcome | UI State | Required Data Shape |
|-------------|----------|---------------------|
| Owner list success | Share management list | Safe metadata only; no credentials, raw tokens, hashes, limiter state, owner identity, backup data, or unrelated vault data |
| Owner single create success | One-time result dialog | Safe metadata plus raw public URL/token and raw access code exactly once |
| Owner batch create success | One-time batch result dialog | Per-account success rows with safe metadata plus raw public URL/token and raw access code exactly once |
| Owner batch partial failure | Mixed result dialog | Success rows keep one-time secrets until copied/closed; failed rows expose only owner-safe item labels and generic failure |
| Owner revoke success | Revoked row and success message | Safe metadata with `status: "revoked"` and `revokedAt`; include revocation limitation copy |
| Owner latest-share-wins | Active row replacement | Backend has made previous active links inaccessible before UI shows the new link as current |
| Recipient unlock success | Shared item view | `SharedItemView` allowlist only |
| Recipient inaccessible | Generic unavailable | Same public state for not found, expired, revoked, deleted, locked, wrong code, malformed token, and cleanup-affected cases |

### Accessibility And Privacy Rules

| Rule | Contract |
|------|----------|
| Keyboard access | Navigation, row actions, filters, copy controls, revoke confirmation, and batch result dialog are keyboard reachable. |
| Focus management | One-time result dialogs trap focus while open and return focus to the triggering Share button after close. |
| Status accessibility | Every status has visible text and an accessible name; color is never the only status signal. |
| Destructive confirmation | Revoke confirmation uses the destructive confirmation copy and requires explicit confirmation. |
| Secret copy feedback | Feedback says `Copied`; it must not echo the copied secret. |
| Screen-reader privacy | Hidden accessible text must not reveal details withheld visually, including public failure causes or raw secrets after dialog close. |
| Browser storage | Do not persist raw share token, access code, share URL, password, OTP code, OTP seed, or shared item response in local storage, IndexedDB, service-worker cache, or query params. |
| Client logging | Do not log full share URLs, access codes, passwords, OTP codes, OTP seeds, token hashes, access-code hashes, limiter keys, owner IDs, or internal diagnostics from share flows. |
| Offline behavior | Do not queue share creation, batch sharing, revoke, or recipient unlock in offline sync. |

    Update `src/shared/middleware/auth.ts` so `authMiddleware` supports two credential sources:

    1. Cookie mode:
       - Read `auth_token` from `getCookie(c, 'auth_token')`.
       - If present, require `csrf_token` from `getCookie(c, 'csrf_token')`.
       - Require `c.req.header('X-CSRF-Token')` to exactly equal the `csrf_token` cookie.
       - On mismatch, throw `new AppError('csrf_mismatch', 403)`.

    2. Bearer mode:
       - If no `auth_token` cookie is present, read `c.req.header('Authorization')`.
       - Accept only a value that matches `Bearer <token>` case-insensitively for the scheme and captures a non-empty token string.
       - Do not require CSRF cookie/header in Bearer mode.
       - If neither cookie nor Bearer token is present, throw `new AppError('no_session', 401)`.

    After selecting the token, keep the existing verification flow:
    - `const payload = await verifySecureJWT(token, c.env.JWT_SECRET);`
    - require `payload && payload.userInfo`, otherwise `new AppError('token_expired', 401)`.
    - require `payload.sessionId`, otherwise `new AppError('session_invalid_schema', 401)`.
    - instantiate `new SessionService(c.env)` and call `validateSession(sessionId)`.
    - if validation is false, throw `new AppError('session_kicked_out', 401)`.
    - set `c.set('user', payload.userInfo)` and `c.set('sessionId', sessionId)`.

Mock `hono/cookie`, `@/shared/utils/crypto`, and `@/features/auth/sessionService` so the tests can assert exact credential selection, CSRF behavior, JWT verification, session validation, and `next()` invocation without needing a live database.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| automation client -> `/api/vault/import` | Caller controls headers and import payload, including potentially secret-bearing otpauth content. |
| `Authorization` header -> auth middleware | Bearer credentials are manually supplied and must not be confused with browser cookies. |
| auth middleware -> vault route | Middleware sets trusted `user` and `sessionId` context variables after JWT and DB session validation. |
| vault route -> `VaultService.importAccounts()` | Route passes user identity and raw import content into existing parsing/encryption logic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T5-01 | Spoofing | Bearer JWT auth | mitigate | Verify Bearer tokens with `verifySecureJWT(token, c.env.JWT_SECRET)` and require `payload.userInfo`. |
| T5-02 | Elevation of Privilege | Kicked-out or expired sessions | mitigate | Require `payload.sessionId` and `SessionService.validateSession(sessionId)` before setting `user`. |
| T5-03 | Tampering | Import request body | mitigate | Reuse `VaultService.importAccounts()` so supported formats go through existing normalization, duplicate detection, and secret validation. |
| T5-04 | Information Disclosure | Import route response | mitigate | Return only `{ success, count, duplicates, pending }`; tests must reject `secret`, `vault`, `userInfo`, `sessionId`, or internal IDs in the response. |
| T5-05 | CSRF | Browser cookie auth | mitigate | Cookie mode continues requiring double-submit CSRF; only manually supplied Bearer auth skips CSRF. |
</threat_model>

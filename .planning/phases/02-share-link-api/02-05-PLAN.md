<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser origin -> `/api/*` | Untrusted websites can send credentialed browser requests; CORS controls whether JavaScript can read authenticated API responses. |
| Public recipient -> `resolveShareAccess()` | Untrusted token/code submissions reach share lookup and must not trigger secret processing until possession of the independent access code is verified. |
| Vault secret -> SharedItemView | Encrypted vault secret material crosses into memory only to generate allowlisted recipient output after access authorization succeeds. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05-01 | Information Disclosure | `src/app/index.ts` CORS middleware | mitigate | Replace reflected credentialed origin callback with exact configured-origin allowlist via `resolveApiCorsOrigin()`, and add tests proving arbitrary origins return `null`. |
| T-02-05-02 | Elevation of Privilege | Owner `/api/share` APIs | mitigate | Preserve `authMiddleware`, CSRF/session validation, and owner-scoped service calls from completed 02-02 while preventing cross-origin credentialed reads of their responses. |
| T-02-05-03 | Information Disclosure | `ShareService.resolveShareAccess()` | mitigate | Move `verifyShareSecret()` before `decryptField()` and OTP `generate()`; add regression spies proving wrong-code requests do not process vault secret material. |
| T-02-05-04 | Repudiation | Share access audit events | mitigate | Preserve existing `access_granted` audit only after successful code verification and secret processing; wrong-code attempts continue to use generic inaccessible response without successful access audit. |
</threat_model>

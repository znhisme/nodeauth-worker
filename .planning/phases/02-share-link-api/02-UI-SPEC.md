# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 2 remains API-only in this checkout. Editable frontend source is not present; only generated `frontend/dist/**` Vue/Vite/Element Plus assets are available. Do not plan direct edits to built or minified frontend assets as the implementation path.

This phase must still shape the future UI by returning safe, display-ready API states for owner share management and public recipient access:

| Surface | Required Future Interaction Contract | Phase 2 API Contract |
|---------|--------------------------------------|----------------------|
| Owner share creation | Owner starts from exactly one vault/account item and creates one bounded share link. | Create endpoint accepts one owner-accessible item reference plus bounded expiration inputs. |
| One-time secret handoff | Owner sees raw share URL/token and access code exactly once, with separate copy actions. | Create response returns raw token/share URL and raw access code only in the create success response. |
| Owner share list | Owner scans active, expired, revoked, and accessed shares without seeing credentials. | List response returns safe metadata: item label/reference, status, created time, expiration, revocation time, last access time, and access count. |
| Owner share detail | Owner inspects one share for status and audit-oriented metadata only. | Detail response omits raw token, raw access code, token hash, access-code hash, password, OTP seed, OTP code, and internal owner/session fields. |
| Owner revoke flow | Owner revokes one share after confirming that revocation only blocks future access. | Revoke endpoint is destructive, owner-scoped, immediate, and returns safe status metadata only. |
| Recipient access-code gate | Friend opens a public link without a NodeAuth account and submits the access code outside the URL. | Public recipient access uses a non-authenticated POST body such as `{ "accessCode": "..." }`; code in query strings is unsupported. |
| Recipient shared item view | Friend sees only the shared account view and copy/reveal controls for allowed fields. | Public success response returns only the `SharedItemView` allowlist: service, account, optional password, optional current OTP code/countdown. |
| Recipient inaccessible state | Missing, invalid, expired, revoked, deleted, locked, and wrong-code states look identical. | Public failure response uses the same generic unavailable state and no-store/no-referrer headers for all inaccessible cases. |

| Element | Copy |
|---------|------|
| Primary CTA | Create share link |
| Empty state heading | No share links yet |

| Error state | This share link is unavailable. Check the link and access code, then ask the owner for a new link if needed. |
| Destructive confirmation | Revoke share link: Revoking this link blocks future access, but it cannot retract credentials already viewed or copied. |

| Expiration label | Expires |
| Access-code label | Access code |
| Recipient unlock CTA | Unlock shared item |
| Copy link action | Copy link |
| Copy access-code action | Copy access code |
| Copy password action | Copy password |
| Copy OTP action | Copy code |
| Copy success feedback | Copied |

## Interaction Contract

### Visual Hierarchy

Future owner screens must put the one-time handoff warning above the raw link and access code immediately after creation. The link and access code must be shown as two separate rows with independent copy controls. Future recipient screens must put the access-code form first; after unlock, the shared account identity, password copy/reveal controls, and current OTP code/countdown become the primary focal points.

### Owner Flow

1. Owner starts from exactly one vault/account item.
2. Owner cannot create a share without bounded expiration and default independent access-code protection.
3. Create success displays raw share URL/token and raw access code only once.
4. Link copy and access-code copy are separate actions; do not combine them into one clipboard action by default.
5. Owner list and detail views show safe metadata only: item label/reference, status, created time, expiration time, revoked time, last accessed time, and access count.
6. Owner inspect never reveals raw token, raw access code, token hash, access-code hash, password, current OTP code, OTP seed, owner email, session data, backup data, or raw internal IDs not needed by the UI.
7. Revoke is a destructive action that requires the confirmation copy declared in this spec.

### Recipient Flow

1. Recipient can open the public link without an authenticated NodeAuth session.
2. Access code is entered into an input and submitted in the request body or equivalent non-URL channel.
3. Recipient failure states use the generic unavailable copy for missing, invalid, expired, revoked, deleted, locked, and wrong-code cases.
4. Recipient success displays only fields from `SharedItemView`.
5. Password and OTP values require explicit reveal/copy controls if visually hidden by default.
6. TOTP display shows the current code plus countdown; raw seed, otpauth URI, algorithm internals, and future-code generation material are not shown.
7. Public recipient access is online-only; do not cache the shared item for offline use.

### API Response Semantics For UI

| API Outcome | Future UI State | Required Data Shape |
|-------------|-----------------|---------------------|
| Owner create success | One-time handoff | `publicUrl` or raw token, raw access code, status, expiresAt, createdAt, safe item label/reference |
| Owner list success | Share management list | Array of safe metadata only; no credentials or hashes |
| Owner inspect success | Share detail | One safe metadata record only; no credentials or hashes |
| Owner revoke success | Revoked confirmation | Safe metadata with `status: "revoked"` and `revokedAt` |
| Recipient unlock success | Shared item view | `SharedItemView` allowlist only |
| Recipient inaccessible | Generic unavailable | Same public error envelope for not found, expired, revoked, deleted, locked, wrong code, and malformed token |

### Security And Privacy UI Rules

| Rule | Contract |
|------|----------|
| Browser storage | Future frontend must not persist raw share token, access code, password, OTP code, OTP seed, or share URL in local storage, IndexedDB, service-worker cache, or query params. |
| Clipboard feedback | Copy feedback says only `Copied`; it must not echo the copied secret. |
| Logs/analytics | Future UI must not send full share URLs, access codes, passwords, OTP codes, OTP seeds, token hashes, access-code hashes, or internal IDs to analytics or client logs. |
| Referrer risk | Recipient views rely on backend `Referrer-Policy: no-referrer`; any future external links from a recipient page must use `rel="noreferrer"`. |
| Cache behavior | Recipient responses rely on backend `Cache-Control: no-store` and `Pragma: no-cache`; future UI must not override this with service-worker caching. |

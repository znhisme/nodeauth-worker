<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner route -> service | Authenticated request data becomes share creation/list/detail/revoke inputs. |
| service -> repository | Owner IDs and share IDs become database filters. |
| service -> recipient DTO | Decrypted vault secret material becomes allowlisted public JSON. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Elevation of Privilege | `ShareRepository.listForOwner`, `ShareService.getShareForOwner` | mitigate | Every owner metadata method filters by `ownerId` from auth context; no method accepts ownerId from request body. |
| T-02-01-02 | Information Disclosure | `OwnerShareMetadataView` and `OwnerShareCreatedView` | mitigate | DTO allowlists omit `tokenHash`, `accessCodeHash`, owner/session fields, password, OTP seed, and current OTP code except create returns raw token/code exactly once. |
| T-02-01-03 | Information Disclosure | `SharedItemView.otp` | mitigate | Service decrypts seed only in memory, generates current OTP, returns only `{ code, period, remainingSeconds }`, and tests assert seed/otpauth absence. |
| T-02-01-04 | Tampering | `revokeShareForOwner` | mitigate | Revoke uses `revokeForOwner(shareId, ownerId, now)` and returns metadata only after owner-scoped read confirms the share. |
| T-02-01-05 | Repudiation | Share audit events | mitigate | Existing Phase 1 `created`, `revoked`, `expired`, and `access_granted` audit writes remain below routes and tests preserve safe metadata only. |
</threat_model>

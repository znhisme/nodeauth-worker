};

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public recipient -> share middleware | Unauthenticated requests provide headers before rate-limit enforcement and token/code processing. |
| proxy/platform -> Hono request headers | Cloudflare, Docker reverse proxies, and Netlify can supply different client identifier headers. |
| limiter key -> persistent database row | Derived bucket keys are stored in `share_rate_limits` and affect future recipient access. |
| source -> generated runtime bundles | Deployment uses generated Worker, Docker, and Netlify backend artifacts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T3-26 | Spoofing | forwarded client headers in `resolveShareRateLimitClientIp` | accept | Forwarded headers can be spoofed in direct Docker/self-host paths without a trusted proxy policy. This gap closure does not treat the identifier as identity; it only avoids collapsing available runtime-provided identifiers into one shared `unknown` bucket. No authorization or audit attribution may depend on this value. |
| T3-27 | Denial of Service | shared unknown limiter bucket | mitigate | Use `CF-Connecting-IP`, first `x-forwarded-for`, `x-real-ip`, `x-nf-client-connection-ip`, then `client-ip` before falling back to `unknown`; add tests proving Docker/Netlify-style headers do not use `share:unknown:share-public-access:<tokenHash>`. |
| T3-28 | Denial of Service | rate-limit bypass through arbitrary forwarded values | accept | Public share links still require high-entropy token plus access code, and this plan preserves fail-closed repository enforcement. Header spoofing can spread attempts across buckets on direct untrusted deployments; solving trusted proxy enforcement is outside this verification gap and must not block closing the cross-runtime collapse bug. |
| T3-29 | Information Disclosure | persisted limiter keys and logs | mitigate | Store only bucket component plus HMAC token hash; do not log client identifiers; keep denied audit `ipHash: null` and `userAgentHash: null`; tests must assert raw token, URL, access code, password, OTP seed, and share IDs are absent from limiter/audit data. |
| T3-30 | Repudiation | recipient attribution from forwarded headers | accept | Limiter keys are operational controls, not reliable identity evidence. The plan must not expose or document them as audit attribution. |
| T3-31 | Tampering | generated bundle drift | mitigate | Rebuild Worker, Docker, and Netlify bundles with project scripts only; verify source maps and grep generated bundles for resolver/header markers. |
| T3-32 | Information Disclosure | public failure response regression | mitigate | Preserve `share_inaccessible` generic response and no-store/no-referrer headers; generated assertions require `share_inaccessible`, `Cache-Control`, and `Referrer-Policy`. |
</threat_model>

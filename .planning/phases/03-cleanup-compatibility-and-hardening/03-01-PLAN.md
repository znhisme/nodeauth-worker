<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduler/opportunistic runtime -> cleanup service | Runtime hooks trigger cleanup with no user request context. |
| cleanup service -> repository | Service must receive counts and avoid leaking stored share/link/limiter row details. |
| public recipient -> expired share access | Cleanup must not make expired/deleted/revoked/wrong-code states distinguishable. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T3-01 | Information Disclosure | `cleanupShareState()` | mitigate | Return count-only `ShareCleanupResult`; tests grep/serialize output for absence of IDs, tokens, codes, owner identity, labels, passwords, seeds, and URLs. |
| T3-02 | Repudiation | expired audit cleanup | mitigate | `insertExpiredAuditEventIfMissing()` records one safe `expired` audit event per share with status/timestamps only. |
| T3-03 | Denial of Service | `share_rate_limits` table | mitigate | `deleteStaleRateLimits(cutoff)` removes stale limiter rows by `lastAttemptAt` without touching active windows/locks. |
| T3-04 | Information Disclosure | expired/revoked/deleted public states | mitigate | Preserve generic inaccessible decisions and public headers; tests assert public semantics continue after cleanup additions. |
</threat_model>

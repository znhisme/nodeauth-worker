<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| platform scheduler -> application service | Worker cron and Docker cron invoke cleanup outside a user request. |
| Netlify request -> opportunistic maintenance | Public or authenticated traffic may trigger internal cleanup after the interval guard. |
| cleanup result -> logs | Runtime logs must expose counts only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T3-05 | Denial of Service | Worker scheduled handler | mitigate | Use `ctx.waitUntil(Promise.all(...))` so cleanup does not replace backup and scheduled failures stay asynchronous. |
| T3-06 | Denial of Service | Docker cron | mitigate | Keep existing backup cron and wrap share cleanup in its own try/catch. |
| T3-07 | Information Disclosure | runtime cleanup logs | mitigate | Log only `expiredSharesMarked` and `staleRateLimitRowsDeleted`; tests grep runtime sources for cleanup logging markers. |
| T3-08 | Information Disclosure | Netlify opportunistic cleanup | mitigate | Hourly guard and generic failure logging prevent per-request detail leakage while preserving request handling. |
</threat_model>

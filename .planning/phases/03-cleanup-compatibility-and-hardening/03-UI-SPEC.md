## Phase UI Scope

Phase 3 is API/docs/backend-test only in this checkout. Editable frontend source is still absent; only generated `frontend/dist/**` Vue/Vite/Element Plus assets are present. Do not plan direct edits to built or minified frontend assets as the implementation path.

This phase must lock the owner-facing and recipient-facing semantics that future UI will consume while Phase 3 implementation updates backend cleanup behavior, API/docs wording, compatibility checks, and hardening tests:

| Surface | Required Future Interaction Contract | Phase 3 Output |
|---------|--------------------------------------|----------------|
| Owner share management list | Owner can distinguish active, expired, revoked, and accessed shares from safe metadata only. Expired cleanup must not make status ambiguous. | API/docs/tests preserve `active`, `expired`, and `revoked` meanings and safe metadata allowlists. |
| Owner revoked state | Revoked means future recipient access is blocked. It never means copied credentials were recalled. | Docs/API notes must say revocation stops future access but cannot retract credentials already viewed or copied. |
| Owner expired state | Expired means the link is no longer usable because its TTL ended. Cleanup may mark/audit/remove stale internal state without changing this user-facing meaning. | Cleanup behavior must keep expired shares inactive and safely explainable. |
| Owner accessed signal | `lastAccessedAt` and `accessCount` indicate prior successful access and copied-risk potential, not proof that credentials were or were not copied. | API/docs must avoid overclaiming access certainty. |
| Recipient unavailable state | Missing, invalid, expired, revoked, deleted, locked/rate-limited, wrong-code, and cleanup-affected shares all appear as the same generic public state. | Public API responses and docs keep generic wording and no-store/no-referrer protections. |
| Stale rate-limit cleanup | Cleanup of old limiter rows is internal maintenance. It must not create recipient-visible or owner-visible history, identity, token, or security detail leaks. | Runtime cleanup logs expose counts only, never share IDs, raw tokens, code hashes, item labels, owner identity, or recipient data. |
| Compatibility status | Supported deployments must present the same share status semantics on Cloudflare Workers, Docker, and Netlify. | Docs/tests assert cross-runtime behavior without platform-specific UI promises. |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review share status |
| Empty state heading | No share links yet |
| Empty state body | Create a link from a single account item when someone needs temporary access. |
| Error state | This share link is unavailable. Check the link and access code, then ask the owner for a new link if needed. |
| Destructive confirmation | Revoke share link: Revoking this link blocks future access, but it cannot retract credentials already viewed or copied. |

Additional required copy:

| Location | Copy |
|----------|------|
| Owner revocation explanation | Revocation stops future access to this link. It cannot retract passwords, OTP codes, or account details already viewed or copied by a recipient. |
| Owner revoked status help | This link has been revoked and cannot be used again. Any credentials already copied by a recipient remain outside NodeAuth's control. |
| Owner expired status help | This link has expired and cannot be used again. Expiration does not remove credentials already copied before it expired. |
| Owner accessed warning | This link has been accessed before. Treat the shared credentials as potentially copied. |
| Cleanup status note | Expired shares and stale access-attempt records may be cleaned up automatically. Cleanup does not change whether copied credentials remain usable outside NodeAuth. |
| Recipient generic inaccessible heading | Share link unavailable |
| Recipient generic inaccessible body | This link may be expired, revoked, locked, or the code may be incorrect. Ask the owner for a new link if needed. |
| Recipient locked/rate-limited state | Use the generic inaccessible heading and body above; do not mention lockout, attempts, token validity, or rate-limit details. |
| Copy success feedback | Copied |
| Owner revoke success | Share link revoked |

Public copy must never reveal whether the token exists, whether the access code was wrong, whether a vault item was deleted, whether the link was revoked, whether the link expired, whether cleanup has run, or whether rate limiting is active.

## Interaction Contract

### Visual Hierarchy

Future owner screens must make revocation limitations visible anywhere revoke is presented, not only in documentation. Status rows must prioritize current status, expiration, last accessed time, and access count. Cleanup or compatibility maintenance text belongs below user-facing status and must never be the primary visual focus.

Future recipient screens must continue to put the access-code form first. Any unavailable state, including cleanup-affected, expired, revoked, deleted, locked/rate-limited, missing-token, and wrong-code cases, must render the same generic unavailable state.

### Owner Flow

1. Owner list and detail views show safe metadata only: item label/reference, status, created time, expiration time, revoked time, last accessed time, and access count.
2. Owner inspect never reveals raw token, raw access code, token hash, access-code hash, password, current OTP code, OTP seed, owner email, session data, backup data, cleanup internals, limiter rows, or raw internal IDs not needed by the UI.
3. `status: "active"` means the link can currently be used if the recipient has the URL and access code.
4. `status: "expired"` means the link can no longer be used because its TTL ended; cleanup may add audit/maintenance state but must not create a new public status.
5. `status: "revoked"` means the owner blocked future use; it does not promise retroactive secret recall.
6. `lastAccessedAt` and `accessCount` must be presented as evidence of successful access only, not as proof of whether credentials were copied.
7. Revoke is a destructive action that requires the confirmation copy declared in this spec.

### Recipient Flow

1. Recipient can open the public link without an authenticated NodeAuth session.
2. Access code is entered into an input and submitted in the request body or equivalent non-URL channel.
3. Recipient failure states use the generic unavailable copy for missing, invalid, expired, revoked, deleted, locked/rate-limited, wrong-code, cleanup-affected, and malformed-token cases.
4. Recipient success displays only fields from `SharedItemView`.
5. Password and OTP values require explicit reveal/copy controls if visually hidden by default.
6. TOTP display shows the current code plus countdown; raw seed, otpauth URI, algorithm internals, and future-code generation material are not shown.
7. Public recipient access is online-only; do not cache the shared item for offline use.

### Cleanup And Status Semantics

| Backend Condition | Owner-Facing State | Recipient-Facing State | Required Wording Constraint |
|-------------------|-------------------|-------------------------|-----------------------------|
| Share active and not expired | Active | Access-code gate or shared item after successful unlock | Do not imply no recipient has copied credentials. |
| Share expired by TTL | Expired | Generic unavailable | Say the link cannot be used again; do not distinguish expired publicly. |
| Share revoked by owner | Revoked | Generic unavailable | Say future access is blocked and copied credentials cannot be retracted. |
| Vault item deleted | Existing safe metadata if available; otherwise omit item detail safely | Generic unavailable | Do not expose deletion publicly. |
| Wrong access code | No owner-facing state change except safe audit/counts if exposed later | Generic unavailable | Do not say the code was wrong in public UI. |
| Rate limited or locked | No owner-facing state unless future safe audit summary is explicitly designed | Generic unavailable | Do not mention attempts, lock timing, or token validity publicly. |
| Stale limiter cleanup ran | No user-facing state; optional docs/admin maintenance summary only | No visible state | Logs/docs may show counts only. |
| Expired-share cleanup ran | Status remains expired or safely removed from active views by policy | Generic unavailable | Cleanup cannot imply copied credentials were recalled. |

### API Response Semantics For UI

| API Outcome | Future UI State | Required Data Shape |
|-------------|-----------------|---------------------|
| Owner list success | Share management list | Array of safe metadata only; no credentials, raw tokens, hashes, limiter state, or cleanup row details |
| Owner inspect success | Share detail | One safe metadata record only; no credentials, raw tokens, hashes, limiter state, or cleanup row details |
| Owner revoke success | Revoked confirmation | Safe metadata with `status: "revoked"` and `revokedAt` plus docs/copy that revocation only blocks future access |
| Expired cleanup success | Maintenance completion in logs/docs only | Counts only, such as expired shares marked/audited and stale limiter rows removed |
| Recipient unlock success | Shared item view | `SharedItemView` allowlist only |
| Recipient inaccessible | Generic unavailable | Same public error envelope for not found, expired, revoked, deleted, locked, wrong code, malformed token, and cleanup-affected cases |

### Accessibility And Privacy Rules

| Rule | Contract |
|------|----------|
| Status accessibility | Every status must have a text label and accessible name; never rely on color alone. |
| Destructive confirmation | Revoke confirmation must be keyboard reachable, focus trapped while open, and announced with the revocation limitation copy. |
| Public errors | Generic unavailable state must be announced as one error state and must not reveal hidden condition details through screen-reader-only text. |
| Browser storage | Future frontend must not persist raw share token, access code, password, OTP code, OTP seed, or share URL in local storage, IndexedDB, service-worker cache, or query params. |
| Clipboard feedback | Copy feedback says only `Copied`; it must not echo the copied secret. |
| Logs/analytics | Future UI must not send full share URLs, access codes, passwords, OTP codes, OTP seeds, token hashes, access-code hashes, cleanup row data, limiter keys, or internal IDs to analytics or client logs. |
| Referrer risk | Recipient views rely on backend `Referrer-Policy: no-referrer`; any future external links from a recipient page must use `rel="noreferrer"`. |
| Cache behavior | Recipient responses rely on backend `Cache-Control: no-store` and `Pragma: no-cache`; future UI must not override this with service-worker caching. |

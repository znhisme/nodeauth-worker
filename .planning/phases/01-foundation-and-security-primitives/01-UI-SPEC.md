# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase UI Scope

Phase 1 is API-only/documentation-only in this checkout. Editable frontend source is not present; only generated `frontend/dist/**` Vue/Vite/Element Plus assets are available. Do not promise or plan direct edits to built or minified frontend assets as the implementation path.

UX-04 is satisfied in Phase 1 by defining the owner and recipient UI surfaces future phases must support if editable frontend source is restored:

| Surface | Required Future Interaction Contract | Phase 1 Output |
|---------|--------------------------------------|----------------|
| Owner account action | Owner can start sharing from one vault/account item only. The entry point must be item-scoped, not vault-scoped. | API/security contract must expose one-item share creation inputs. |
| Share creation form | Owner chooses bounded expiration and receives/generated independent access code by default. Primary action creates the share. | API/security contract must require expiration and access code policy. |
| One-time secret handoff | Owner sees raw share URL token/link and access code exactly once, with separate copy actions and clear warning that values cannot be recovered later. | API contract must return raw token/code only on creation. |
| Share management list | Owner can distinguish active, expired, revoked, and accessed links from safe metadata only. | Schema/service contract must store safe status, timestamps, and counts. |
| Revoke flow | Owner can revoke one link with a destructive confirmation that says revocation blocks future access but cannot retract copied credentials. | Service contract must enforce revocation below routes. |
| Recipient access-code gate | Friend opens link without NodeAuth account and enters access code through a form body, never through URL query. | Recipient API contract must require non-URL access-code submission. |
| Recipient shared item view | Friend sees only the shared item, with reveal/copy controls for allowed fields and current TOTP code/countdown when available; raw OTP seed is not shown. | DTO contract must define minimal `SharedItemView` allowlist. |
| Recipient inaccessible state | Missing, invalid, expired, revoked, deleted, locked, and wrong-code cases all use the same generic public state. | API contract must use generic public errors and no-store/no-referrer protections. |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Create share link |
| Empty state heading | No share links yet |
| Empty state body | Create a link from a single account item when someone needs temporary access. |
| Error state | This share link is unavailable. Check the link and access code, then ask the owner for a new link if needed. |
| Destructive confirmation | Revoke share link: Revoking this link blocks future access, but it cannot retract credentials already viewed or copied. |

Additional required future copy:

| Location | Copy |
|----------|------|
| One-time handoff warning | Save this link and access code now. NodeAuth will not show them again. |
| Access-code label | Access code |
| Recipient unlock CTA | Unlock shared item |
| Copy link action | Copy link |
| Copy access-code action | Copy access code |
| Copy password action | Copy password |
| Copy OTP action | Copy code |
| Recipient generic inaccessible heading | Share link unavailable |
| Recipient generic inaccessible body | This link may be expired, revoked, locked, or the code may be incorrect. |

Copy must never expose whether the token exists, whether the access code was wrong, whether a vault item was deleted, or whether rate limiting is active.

## Interaction Contract

### Owner Flow

1. Owner starts from exactly one vault/account item.
2. Owner cannot create a share without a bounded expiration and independent access code policy.
3. Share creation returns the raw link token/share URL and access code only in the success state immediately after creation.
4. Owner copy controls are separate: link and access code must not be combined into one automatic clipboard action.
5. Share-management views show safe metadata only: item label/reference, status, created time, expiration time, last accessed time, and access count.
6. Inspecting a share never reveals raw token, raw access code, password, TOTP seed, or recipient credential values.
7. Revocation is a destructive action with explicit confirmation copy from this spec.

### Recipient Flow

1. Recipient can open the public link without an authenticated NodeAuth session.
2. Access code is entered into an input and submitted in a request body or equivalent non-URL channel.
3. Access-code input uses 16px text on mobile to avoid viewport zoom, masks only if future security review prefers masking, and provides a 44px minimum submit target.
4. Recipient view displays only the shared account fields in the `SharedItemView` allowlist.
5. Password and OTP values require explicit reveal/copy actions if they are visually hidden by default.
6. TOTP display shows current code plus countdown; raw seed and otpauth URI are not shown by default.
7. All inaccessible states collapse to the generic error copy above.

### Security And Privacy UI Rules

| Rule | Contract |
|------|----------|
| Browser storage | Future frontend must not persist raw share token, access code, password, or OTP seed in local storage, IndexedDB, service-worker cache, or query params. |
| Clipboard feedback | Copy feedback must confirm only the action, such as "Copied", not repeat the copied secret. |
| Logs/analytics | Future UI must not send full share URLs, access codes, passwords, OTP codes, OTP seeds, or internal IDs to analytics or client logs. |
| Referrer risk | Recipient pages must rely on backend no-referrer/no-store headers; future UI links out from recipient view should use `rel="noreferrer"` if any external links are added. |
| Offline mode | Public share access is online-only; do not cache recipient shared item data for offline use. |
